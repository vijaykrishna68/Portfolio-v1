## Problem

The platform had 200+ enterprise customers, each with their own API key scheme, rate limit tiers, and routing requirements. The existing gateway was a Node.js service that was fast enough at 1M requests/day. At 12M it started dropping requests under burst load.

Two constraints drove the rewrite:

1. **Latency budget.** The P99 gateway overhead could not exceed 10ms. Customer SLAs were written against end-to-end latency including the gateway.
2. **Fairness.** One customer running a bulk sync job shouldn't degrade the experience for a customer running a real-time integration.

## Rate Limiting Design

We evaluated token bucket and leaky bucket algorithms before landing on a **sliding window counter** in Redis.

The sliding window algorithm maintains a per-customer counter that tracks requests in the last N seconds exactly. No burst spikes at window boundaries (a problem with fixed-window counters). Acceptable memory footprint (two sorted sets per customer).

The key insight for fairness: rate limits are defined per customer, but **burst budgets** are separate. A customer with a 1,000 req/min limit gets a burst budget of 200 requests that can be spent in any 5-second window. This lets legitimate low-latency integrations spike without triggering limits, while preventing bulk jobs from consuming headroom.

The Redis EVAL script that implements the check-and-increment is atomic. At 12M requests/day the Redis cluster handles roughly 140 requests/second — well within capacity.

## Why Rust

Rust was chosen for one reason: we needed zero-copy request parsing on the hot path.

The JWT validation path was the bottleneck in the Node.js implementation. Each request required: base64 decode the token, parse the JSON payload, verify the HMAC signature, check the customer ID against a Redis lookup. In Node.js this involved 4 heap allocations per request. In Rust, we parse directly from the raw byte buffer with no intermediate allocations.

The resulting binary uses about 4MB of RSS at full load. The Node.js service used 200MB.

## Observability

Every request emits four Prometheus counters: `requests_total`, `requests_by_customer`, `rate_limited_total`, `latency_bucket`. The Grafana dashboard shows per-customer traffic in real time, making it immediately visible when a customer is hitting limits or behaving anomalously.

The one Prometheus label we deliberately avoided: customer name on the `latency_bucket` histogram. With 200 customers and 50 latency buckets, that's 10,000 time series. We track customer latency via a sampled secondary metric written to TimescaleDB.

## Lessons Learned

**The latency budget forced good decisions.** Anything that added more than 0.5ms had to be justified explicitly. This prevented scope creep: we didn't add request body inspection, we didn't add response transformation, we didn't add per-endpoint rate limits. The constraint kept the gateway doing exactly one thing well.

**Redis cluster failover was harder than expected.** The sliding window algorithm requires strong consistency — a counter miss means a rate limit bypass. We spent two weeks on the Redis cluster failover path to ensure the gateway fails closed (blocks requests) rather than open (allows unlimited requests) during a Redis partition event.
